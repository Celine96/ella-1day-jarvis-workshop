# 부록

워크숍이 끝난 뒤에 펼쳐보는 자가 학습 자료예요. 막힌 게 있을 때, 권한을 정리하실 때, 채널을 추가하실 때, 보안을 점검하실 때 — 본인 상황에 맞는 섹션만 골라 보시면 됩니다.

| 섹션 | 언제 펼쳐보세요 |
|---|---|
| **A. MCP 트러블슈팅** | routine이 안 작동하거나 권한 에러가 날 때 |
| **B. 권한 회수 가이드** | 자비스 권한을 정리하거나 잠시 멈추실 때 |
| **C. 옵션 채널 레시피** | 워크숍에서 연결하지 않은 다른 채널을 추가하실 때 |
| **D. 자비스 보안 체크리스트** | 매주·매월 정기 보안 점검 |
| **E. 자주 묻는 질문** | 사소한 운영 질문 |
| **F. 에이전트 확장** | Q1 반복 작업·Q3 위임을 에이전트로 자가 자동화 |

## A. MCP 트러블슈팅

### 인증 만료
Google · Slack · Notion 토큰은 일정 기간(보통 6개월) 미사용 시 자동 만료됩니다. routine이 작동하지 않으면 가장 먼저 의심하실 항목이에요.

클로드 코드 터미널에서 다음 슬래시 커맨드를 입력하세요.

**[커맨드 입력]**

```
/2-2-mcp-test
```

테스트가 실패하면 해당 MCP를 재인증. OAuth 동의 흐름을 다시 거치시면 즉시 작동합니다 (2-2와 같은 흐름).

### 권한 거절
OAuth 동의 시 일부 권한을 누락하셨을 가능성이에요.

| 증상 | 누락 가능성 |
|---|---|
| "메일을 보낼 권한이 없습니다" | Gmail 쓰기 권한 미추가 (routine 자기발송용) |
| "캘린더에 접근할 수 없습니다" | Calendar 읽기 권한 미추가 |
| "routine 메일 자기발송 실패" | Gmail 쓰기 권한 미추가 |

해결: 해당 MCP 재인증, OAuth 동의 화면에서 누락된 권한을 추가로 체크.

### 한도 초과 (429)
Google API 일일 한도 초과 시 발생합니다. 일반 사용에선 거의 발생 안 해요. 다음 날까지 대기하시면 자동 해결됩니다.

### routine 실행됐는데 메일이 안 옴
본문 [3-4 "메일이 안 오면"](/part3/3-4-automation#메일이-안-오면)과 같은 흐름으로 점검하시면 됩니다. 가장 흔한 원인은 **Gmail 스팸 분류** — 본인이 본인에게 보내는 메일이라 스팸 폴더로 가는 경우예요.

1. Gmail 스팸 폴더 확인 → "스팸 아님" 표시 + 화이트리스트 등록
2. 발송 로그 확인:
   ```
   /3-4-test-routine-now
   ```
   콘솔 출력에서 어디서 멈췄는지 확인

## B. 권한 회수 가이드

자비스 권한 회수는 채널별로 1분 안에 끝납니다. 회수 즉시 자비스는 본인 데이터에 더 이상 접근할 수 없고, 다음 routine 실행 시점에 자동으로 멈춰요.

### Gmail · Google Calendar
1. Google 계정 → [보안](https://myaccount.google.com/security)
2. **"Google에 액세스할 수 있는 서드파티 앱"** → 클로드 코드 → **액세스 권한 삭제**

### Slack
1. Slack 워크스페이스 → 설정 → **앱 관리**
2. 클로드 코드 검색 → **제거**

### Notion
1. Notion 설정 → **내 연결**
2. 클로드 코드 → **연결 해제**

다시 자비스를 켜고 싶으시면 같은 OAuth 동의를 다시 거치시면 됩니다 (2-2 · 3-1과 같은 흐름). 페르소나 파일과 운영 커맨드는 그대로 살아 있어 재인증만 하시면 즉시 평소대로 작동해요.

## C. 옵션 채널 레시피

자비스에 본인 핵심 채널(Slack · Notion · LinkedIn)을 연결하면 그 채널의 **Q3 위임 영역까지 자비스가 정리**해드려요. 워크숍 본 흐름에는 시간 한정으로 빠져 있어, 본인이 필요할 때 자가 확장하실 수 있도록 가이드를 정리했습니다.

| 직군 | 추천 채널 |
|---|---|
| 팀 운영·실무 중심 | Slack — DM·멘션의 Q3 항목 |
| 문서·OKR·메모 중심 | Notion — OKR·할 일 페이지의 Q3 항목 |
| B2B 사업개발·네트워킹 중심 | LinkedIn — 메시지의 Q3 항목 |

채널당 약 30분이면 끝납니다. 본인 핵심 채널부터 하나씩 추가하세요.

### C-1. Slack 추가 연결 {#slack-recipe}

**용도**: DM·멘션을 사분면으로 분류, `/morning-brief`에 통합

::: details 절차
1. Slack MCP 추가 → 본인 워크스페이스로 OAuth 동의
2. 권한 범위: `channels:read` · `groups:read` · `im:read` · `chat:write`
3. `.claude/commands/`에 `slack-triage.md` 파일 생성:
   ```markdown
   ---
   description: Slack DM·멘션을 아이젠하워 매트릭스로 분류
   ---
   - mcp__slack 으로 채널·DM 미답 메시지 조회 (24h)
   - 페르소나 §4 규칙 적용해 4사분면 분류
   - Q1·Q3 항목에 본인이 처리할 한 줄 태스크 정의 (답장 작성은 안 함)
   ```
4. 페르소나 §1에 "Slack DM·멘션은 /morning-brief에 통합" 한 줄 추가
:::

### C-2. Notion 추가 연결 {#notion-recipe}

**용도**: OKR·할 일 페이지를 `/morning-brief`에 통합

::: details 절차
1. Notion MCP 추가 → 본인 워크스페이스로 OAuth 동의
2. 권한 범위: 읽기 + (옵션) 페이지 생성
3. `.claude/commands/`에 `notion-okr-check.md`:
   ```markdown
   ---
   description: Notion OKR/할 일 페이지를 읽어 morning-brief에 통합
   ---
   - mcp__notion으로 OKR 페이지 검색
   - 미완료 항목을 사분면으로 분류
   ```
4. `/morning-brief`에서 자동 호출되도록 페르소나 §1 한 줄 추가
:::

### C-3. LinkedIn 추가 연결 {#linkedin-recipe}

**용도**: B2B 사업개발 메시지를 사분면으로 분류 + 본인 처리 태스크 정의

::: details 절차
LinkedIn은 공식 MCP가 없어 **수동 텍스트 입력** 패턴입니다.

1. LinkedIn 메시지 텍스트를 주기적으로 추출 (브라우저 확장 또는 수동)
2. `.claude/commands/linkedin-triage.md`:
   ```markdown
   ---
   description: LinkedIn 메시지 텍스트 입력 → 사분면 분류 + 본인 처리 태스크 정의
   ---
   - 입력: LinkedIn 메시지 텍스트
   - 페르소나 §4 규칙으로 사분면 분류
   - Q1·Q3 항목에 본인이 처리할 한 줄 태스크 정의 (답장 작성은 안 함)
   - 결과는 콘솔 출력
   ```
3. 답장 작성·발송은 본인이 직접 LinkedIn에서 처리
:::

## D. 자비스 보안 체크리스트

자비스의 보안 원칙은 [1-2 챕터](/part1/1-2-install#자비스의-보안-원칙)에서 둘로 나눠 다뤘어요 — **시스템에 박혀 있는 안전장치**와 **본인이 직접 챙기시는 것**. 워크숍 종료 후에는 다음 항목들을 정기적으로 점검해두시면 안전 수준이 그대로 유지됩니다.

### D.1 운영 점검 (매주 1회 권장)

::: tip 보안 점검 항목 — 매주
- [ ] 페르소나 파일에 비밀번호·계좌·계약 금액 등 민감 정보 없음
- [ ] Gmail 권한 범위 = 읽기 + 보내기 (불필요한 "수정"·"삭제" 권한 없음)
- [ ] routine이 외부 수신자에게 자동 발송하지 않음 (본인 메일함 자기발송만)
- [ ] 자비스가 답장 초안을 작성·임시보관함 저장하지 않음 (분류·태스크 정의까지만)
- [ ] `.claude/agents/자비스.md`가 GitHub 공개 레포에 올라가지 않음
- [ ] OAuth 토큰·API 키가 `.env` 또는 GitHub Secrets에만 저장되어 있고 코드에 평문 노출 안 됨
- [ ] 자비스가 보낸 routine 메일이 본인 메일로만 도착 (외부 수신자 없음)
:::

### D.2 환경 점검 (월 1회 권장)

::: tip 환경 점검
- [ ] Google 계정 보안 페이지에서 클로드 코드 액세스가 의도한 권한 범위인지 확인
- [ ] 사용 안 하는 옵션 채널(Slack · Notion 등)이 연결된 채로 있는지 확인 — 안 쓰면 회수
- [ ] routine 실행 로그에 비정상 발송 시도 없는지 확인
- [ ] 페르소나 파일의 회사 컨텍스트가 최신인지 (퇴사·이직·M&A 등 반영)
:::

### D.3 즉시 대응 시나리오

D.1·D.2 정기 점검을 꾸준히 하시면 다음 시나리오가 발생할 가능성은 매우 낮아요. 만약의 상황을 위한 비상 절차로 한 번 훑어두시면 됩니다.

| 상황 | 즉시 조치 |
|---|---|
| 자비스가 외부 수신자에게 메일 발송 시도 발견 | 1. Gmail 쓰기 권한 즉시 회수 (Google 보안 페이지) 2. routine 등록 해제 3. 페르소나 §7 점검 |
| 페르소나에 실수로 민감 정보 입력 | 1. 페르소나 파일에서 즉시 삭제 2. GitHub 레포 푸시했다면 `git filter-branch` 또는 새 레포로 마이그레이션 3. 노출 자격증명(비번/토큰) 즉시 변경 |
| 노트북 분실·도난 | Google 계정 보안 페이지에서 모든 클로드 코드 권한 일괄 회수 |
| 회사에서 외부 LLM 사용 금지 정책 발효 | routine 등록 해제 + Google 권한 회수 + 페르소나 파일 삭제 |

## E. 자주 묻는 질문

### Q. 자비스가 매일 보내는 메일을 끄고 싶어요
"daily routine 정지해주세요"라고 자비스에게 말씀하시면 됩니다.

### Q. 출장 중에 자비스가 평소처럼 보내면 어색해요
"이번 주 routine 정지" 또는 "다음 주 월요일까지 정지" 자비스에게 명령.

### Q. 자비스 톤을 바꾸고 싶어요
대화로: "오늘부터 더 친근한 톤으로 보고해주세요" → 페르소나 §3 자동 업데이트.

### Q. 자비스가 잘못 분류해요
"이 메일은 항상 Q3로 분류해주세요" → 페르소나 §4 자동 업데이트. 한 달 쓰시면 분류 정확도 매우 올라갑니다.

### Q. 다른 도구(Outlook · Apple Calendar)도 연결할 수 있나요?
공식 MCP가 있는 도구만 직결 가능. Outlook은 현재 베타. Apple은 미지원. 그 외엔 텍스트 입력 패턴(LinkedIn처럼).

## F. 에이전트 확장 — Q1 반복·Q3 위임 자동화 {#f-에이전트-확장}

자비스는 **분류·태스크 정의·담당자 추천**까지 해드리고, **답장 작성·실행·발송**은 본인이 직접 하시는 구조예요. 인간 우선 원칙을 보존하기 위한 의도적 경계입니다.

워크숍 후 본인이 자비스를 며칠·몇 주 쓰시다 보면, **반복적이고 안전한 일부 작업**은 별도 에이전트가 자동 처리하면 좋겠다는 시점이 옵니다. 이 섹션은 그 시점에 자가 개발하실 수 있는 가이드예요.

### F-1. 자비스 vs 에이전트 — 역할 분담

| 역할 | 자비스 | 에이전트 |
|---|---|---|
| **분류** (Q1·Q2·Q3·Q4) | ✓ | — |
| **태스크 정의** (한 줄 요지) | ✓ | — |
| **담당자 추천** (Q3) | ✓ | — |
| **답장 초안 작성** (외부 수신자 대상) | ❌ 안 함 | ✓ (임시보관함 저장, 페르소나 §8 정책 안에서) |
| **반복 작업 실행** (예: 정기 보고서 생성) | — | ✓ |
| **위임 메일 초안 작성** | — | ✓ (페르소나 §8 정책 안에서, 임시보관함 저장) |
| **외부 자동 발송** | ❌ 절대 금지 | ❌ 절대 금지 (인간 우선 원칙 보존) |

**핵심 1**: 자비스 본체는 답장 초안을 작성하지 않습니다. 답변 톤·관계가 자비스 어휘에 묶이는 걸 막기 위한 의도적 경계예요. 답장 초안 자동 작성을 원하시면 별도 에이전트로 명시적으로 자가 정의하셔야 합니다.

**핵심 2**: 에이전트도 자비스의 인간 우선 원칙을 그대로 준수해야 합니다. 외부 수신자에게 자동 발송하는 동작은 절대 허용하지 않아요. 에이전트가 할 수 있는 건 본인 도구 안에서의 자동 워크플로 (임시보관함 저장·내부 데이터 정리·정기 보고서 생성 등)까지입니다.

### F-2. Q1 반복 작업 자동화 패턴 {#f-1-q1-반복}

자비스가 매일 Q1으로 분류해주는 항목 중 **반복적이고 본인이 결정 비용을 들이지 않는 작업**이 자동화 후보예요.

**예시**:
- 매주 월요일 오전 정기 보고서 (지난 주 KPI·이슈 요약)
- 매월 첫 영업일 재무 마감 알림 메일 (CFO에게 자기발송)
- 매주 금요일 OKR 진행률 Notion 페이지 업데이트

**에이전트 패턴 (Claude Agent SDK)**:
1. 자비스 페르소나 §8 자동화 정책에 정의된 작업 식별
2. 별도 슬래시 커맨드 (`/agent-weekly-kpi-report`) 또는 routine으로 정의
3. 데이터 소스(Calendar·Gmail·Notion 등) MCP 호출
4. 결과를 본인 메일함 자기발송 또는 Notion 페이지 작성

**자비스와의 연결**:
- 자비스가 매일 아침 브리핑에서 "오늘 자동화된 작업이 N건 처리됐어요" 안내
- 학습자는 결과만 검토

### F-3. Q3 위임 에이전트 패턴 {#f-2-q3-위임}

자비스가 회의록에서 Q3로 분류한 위임 항목을 별도 에이전트가 받아 **위임 메일 초안 작성 + Gmail 임시보관함 저장**까지 자동 처리하는 패턴이에요. 외부 발송은 학습자가 직접 누름.

**에이전트 패턴**:
1. `/meeting-recap` 출력의 Q3 항목 (담당자 추천 포함) 수신
2. 페르소나 §8 정책 검증 — 위임 가능한 담당자·항목 패턴인지
3. 페르소나 §3 톤으로 위임 메일 초안 작성
4. **Gmail 임시보관함**에 저장 (외부 발송 X)
5. 학습자에게 "위임 메일 N건 임시보관함 저장 완료. 검토 후 보내주세요" 안내

**핵심 안전 정책**:
- 자동 발송 절대 금지 — 임시보관함까지만
- 페르소나 §8에 정의된 담당자·항목만 자동 처리
- 정의되지 않은 케이스는 자비스의 기본 흐름(담당자 추천만)으로 회귀

### F-4. 페르소나 §8 자동화 정책 {#f-3-페르소나-8}

페르소나 파일(`.claude/agents/{자비스 이름}.md`)에 §8 섹션을 추가해 자동화 정책을 정의합니다.

```markdown
## §8 자동화 정책 (선택)

### Q1 반복 자동화 — 허용 작업
- "매주 월요일 KPI 보고서 자동 생성" — 트리거: 월요일 오전 8시
- "월말 재무 마감 알림 메일 자기발송" — 트리거: 매월 28일

### Q3 위임 에이전트 — 허용 담당자·항목
- 재무팀장: 데이터룸·재무 자료 위임 가능
- 외부 자문 (법무·세무): 계약서·신고 위임 가능
- 영업본부장: 고객사 응대 위임 가능

### 자동화 금지 패턴
- 투자자·이사회 관련 메일 절대 자동 처리 금지 (본인 직접만)
- 채용 최종 결정 관련 절대 자동 처리 금지
- 1억 원 초과 결정 관련 절대 자동 처리 금지
```

학습자가 본인 컨텍스트에 맞춰 정의. 비어 있으면 에이전트는 작동하지 않음 (안전 기본값).

### F-5. 안전 가이드 {#f-4-안전}

에이전트 도입 시 반드시 지켜야 할 경계:

- **외부 자동 발송 절대 금지** — 자비스 인간 우선 원칙과 동일
- **페르소나 §8에 명시되지 않은 작업 자동 처리 금지** — 안전 기본값은 "수동"
- **자동화된 작업 로그 매주 점검** — 의도와 다른 처리가 없는지 확인
- **에이전트가 작동한 결과는 본인 메일함으로 자기발송** — 처리 내역 추적 가능
- **에이전트 권한 회수도 1분 안 가능** — 자비스 권한 회수와 동일 절차

### F-6. 자가 개발 시작 {#f-5-시작}

**필요한 도구**:
- [Claude Agent SDK](https://docs.claude.com/agent-sdk) — 에이전트 정의·실행 인프라
- 클로드 코드 슬래시 커맨드 또는 routine — 트리거
- 페르소나 §8 — 정책 정의

**단계**:
1. 페르소나 §8 자동화 정책 작성 (위 F-4 참고)
2. 자동화하고 싶은 첫 번째 작업 선택 (가장 단순한 Q1 반복부터)
3. 슬래시 커맨드 또는 routine으로 에이전트 정의
4. 1주일 테스트 — 결과 매일 점검
5. 안정 확인 후 다음 작업 추가

**호스트 지원**: 자가 개발 중 막히는 부분이 있으면 워크숍 종료 후에도 호스트에게 문의 가능합니다.
